GPU呈现图的含义

  • Misc

  • Input

    • 处理输入的事件的时间
    • 一般回调事件都在主线程处理，如果耗时较大，应该将回调中的逻辑放在子线程进行
    • RecyclerView 的滑动事件也属于这一范畴，RecyclerView的滑动消耗touch事件，并填充itemView
    • Traceview 或者 Systrace工具可以帮助分析

  • Animation

    • 显示ObjectAnimator, ViewPropertyAnimator和Transitions 的耗时
    • 耗时开销在执行动画属性的变化上，比如一个fling动画，大量的view加载和填充就会很耗时

  • Measurement/Layout

    • view的measure和layout的时间消耗
    • 界面过于复杂，层次嵌套过深会消耗大量的时间
    • Traceview 或者 Systrace工具可以帮助分析

  • Draw

    • view的绘制操作，与各种draw()方法有关的方法的耗时
    • 突然大量的view需要绘制，或者绘制逻辑过于复杂会消耗大量的时间

  • Sync/Upload

    • bitmap从CPU内存转换到GPU内存的耗时，CPU和GPU内存区域不一样，当在将bitmap显示在屏幕上之前，会将bitmap从CPU内存转换到GPU内存，GPU内存中会缓存bitmap节约耗时
    • 所有的资源在draw之前都需要居于GPU中，大量小资源或者少量大资源都会开销大量时间
    • 避免使用过大分辨率的bitmap;下一次同步之前调用prepareToDraw()异步上传bitmap

    在5.0上这一块的颜色时紫色的

  • Issuing Commands

    • 将所有指令列表绘制到屏幕上需要的时间。为了让系统在屏幕上绘制显示列表，它将必要的命令发送到GPU。通常，它通过opengl es api来执行这个动作。这个过程需要一些时间，因为在将命令发送到GPU之前，系统对每个命令执行最终转换和剪切，会在GPU端产生额外的开销

    • 一帧中对大量复杂的需要展示列表进行直接测量。例如，有大量的固定开销的draw操作

      会比canvas.drawPoints(mThousandPointArray);更耗时

      这种差异的产生是因为显示列表尽可能被系统高速缓存。因此，在某些情况下，滚动，转换或动画需要系统重新发送显示列表，但不必从头开始重新创建它，重新获取绘图命令。结果，你可以看到一个高的“Issue commands”栏而不会看到一个高的”Draw commands”栏

  • Processing/Swapping Buffers

    • 当讲所有的显示列表发送给GPU后，系统最终告诉图像驱动已经完成这一帧，此时需要最终将图像呈现在屏幕上
    • GPU的工作和CPU是并行的，系统发出绘制命令给GPU后就执行下一条指令，GPU从队列中取出任务执行。当CPU的发出指令的速度大于了GPU的执行指令的速度，那么CPU就会阻塞直到GPU有空间执行之后的指令。这中情况经常会在Swap Buffers阶段出现，因为这个阶段所有的指令都被提交到GPU了。缓解这个问题的办法只有减少”Issue Commands”中复杂的工作

  • Miscellanenous

    • 除了渲染系统执行其工作所需的时间之外，还有一组额外的工作发生在主线程上，并且与渲染无关。这项工作消耗的时间被报告为misc时间。混杂时间通常表示可能在两个连续的渲染帧之间的UI线程上发生的工作。
    • 如果这个时间很长，那app中存在callback，intent和别的工作应该在子线程执行，可以采用Method Tracing或者Systrace工具追踪执行在主线程的方法
